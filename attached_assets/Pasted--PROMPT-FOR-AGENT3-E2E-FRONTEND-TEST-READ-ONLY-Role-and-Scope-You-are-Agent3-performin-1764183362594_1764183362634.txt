### **Global E2E Frontend Test Methodology (applies to the app you select)**

1.  **Test intent:** Run a comprehensive, read-only E2E verification of the frontend experience. Prioritize discovering non-working buttons and links, navigation issues, routing errors, validation problems, and integration gaps visible from the UI.
2.  **Device/responsive checks:** Validate layout and critical flows at these widths: 360px (mobile), 768px (tablet), 1280px (desktop).
3.  **Browsers (if applicable to your environment):** At minimum, emulate Chromium. If multi-browser emulation is available, also test WebKit and Firefox.
4.  **Navigation and link integrity:**
    * Enumerate and click all visible navigation items, buttons, CTAs, and footer links.
    * For each interaction, capture: page/route, element label, expected action, actual result, final URL, HTTP status, and whether a new tab opens.
    * Record all broken links (404/5xx), dead buttons (no action), misrouted links, and CORS/HTTPS errors.
5.  **Forms and validation:**
    * For every visible form: identify required fields, client-side validation, error messages, success messages, and network responses.
    * Test edge cases: empty required fields, invalid emails, short/long inputs, invalid URLs.
6.  **Auth/session (if applicable to your app):**
    * Test sign-up, login, logout, password reset, session persistence/expiry, error handling.
    * Confirm redirects after login/logout and protected-route enforcement.
7.  **SPA routing/deep-links:**
    * Manually enter key routes in the address bar to verify direct access and correct rendering.
    * Verify back/forward navigation consistency.
8.  **Accessibility (light check):** Keyboard navigation for primary flows, focus states on interactive elements, alt text for major images, form labels, and ARIA attributes on key widgets.
9.  **Performance (light check):** Note first meaningful paint impression, network waterfall issues, oversized assets, console warnings/errors, and any blocking resources.
10. **Cross-app links (if they appear on your app’s UI):** Click through any links that point to other Scholar apps. Confirm correct landing pages and return paths. Record any cross-origin issues.
11. **Evidence:** Include exact element labels/selectors (where visible), full URLs, HTTP status codes, visible error messages, and console log excerpts. If you can capture HAR or screenshots, reference them; otherwise, provide verbose textual details.
